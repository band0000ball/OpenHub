# DESIGN — data.go.jp 文字化け調査・修正（Phase 3.5）

**Sprint**: Phase 3.5
**Date**: 2026-04-01
**Status**: 完了（分岐C: 再現不可 → 調査レポート記録）

---

## 背景・課題

data.go.jp は OpenHub のコアコンテンツの一つ。
Bypass コネクターは e-gov.go.jp API を経由してデータを取得しているが、
レスポンスの文字コード処理が不適切な場合、タイトル・説明文が文字化けし
エンドユーザーがデータセット情報を正しく閲覧できない。

現状では文字化けの有無・再現条件・影響範囲が未確認のため、
まず調査から着手し、原因を特定した上で修正を行う。

---

## 目標・非目標

### 目標
- e-gov.go.jp API レスポンスの文字コードを調査・特定する
- data.go.jp コネクター（`bypass/connectors/datagojp.py`）での文字コード処理を修正する
- data.go.jp のタイトルと説明が Catalog UI 上で日本語として正しく表示されることを確認する

### 非目標
- e-Stat・その他コネクターの文字コード対応
- e-gov.go.jp の仕様変更全般への対応
- Catalog UI の表示デザイン変更

---

## 制約条件

- e-gov.go.jp の API 仕様が非公開または不明な可能性がある
  → 実際にリクエストを送り、レスポンスヘッダー・ボディを観測して判断する
- Bypass は Python / httpx 構成のため、文字コード処理は httpx の挙動に依存する

---

## 調査計画

### Step 1: 現状確認
- `datagojp.py` の実装を読み、現在の文字コード処理を確認する
- e-gov.go.jp API に実際にリクエストを送り、`Content-Type` ヘッダーと文字コードを確認する

### Step 2: 問題の特定
- レスポンスが UTF-8 以外の場合（Shift_JIS 等）→ デコード処理の追加・修正
- レスポンスが UTF-8 だが escape 処理に問題がある場合 → JSON パース前後の処理を確認
- API 側の問題で修正不可の場合 → 調査結果レポートを成果物とする

### Step 3: 修正・テスト
- コネクターを修正し、既存テストを更新
- 文字化けケースをカバーするテストを追加
- Catalog UI で表示を目視確認

## リスクと前提

### R1: e-gov.go.jp の仕様が非公開（最重要）
公式ドキュメントが存在しない・古い場合、文字コード仕様が不明になる。
**対策**: 実際の API レスポンスを観測して仕様を逆引きする。修正不可なら調査レポートで完了とする。

### R2: 環境依存の再現
ローカル環境と Lambda 環境でデコード挙動が異なる可能性がある。
**対策**: テストにデコード処理のユニットテストを追加し、環境差異を吸収する。

---

## 調査結果サマリー

**分岐 C: 再現不可** — 現行実装は正しく動作している。

- e-gov.go.jp は `Content-Type: application/json;charset=utf-8` を返す
- 日本語は `\uXXXX` JSON エスケープ形式で返される（バイト列は ASCII 範囲）
- `response.json()` が `\uXXXX` → Python 文字列に正しく変換する
- 文字化けが再発した場合は Catalog UI の表示処理を確認すること

詳細ログ: `memory/20260401-235900.log`
