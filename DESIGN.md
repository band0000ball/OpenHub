## 背景・課題

現在、OpenHub の検索結果・一覧ページは一部のデータしか表示されていない。
ユーザーは全件を閲覧できず、日常利用に耐える品質を満たしていない。

Phase 3.1〜3.3（認証・DynamoDB・Lambda+Amplify デプロイ）が完了し、
次の優先事項としてページネーションの実装が必要になった。

## リスクと前提

### R1: 上流 API ページネーション仕様の差異
e-Stat と data.go.jp はそれぞれ異なるパラメータ体系を持つ可能性がある。
**対策**: Plan フェーズで各 API ドキュメントを調査し、抽象化レイヤーで吸収する。

### R2: `total` 件数が取得できない
上流が総件数を返さない場合、「次ページ存在判定」方式（nextPage フラグ）にフォールバックする。

### R3: BAN リスク（最重要）
ページ切り替えのたびに上流 API を呼び出すと、レート制限・BAN のリスクがある。
**対策**: ページ単位でキャッシュキーを設計し、同一クエリ＋ページの結果を TTL 内で再利用する。

### R4: 複数ソース横断時のページネーション意味論
`source=None`（全ソース横断）でページネーションを行う場合、同一の `offset` を各ソースに渡す。
これにより 1 ページあたり最大 `limit × ソース数` 件が返る（例: limit=20 で e-Stat 20 件 + data.go.jp 20 件 = 最大 40 件/ページ）。
`total` は各ソースの全件数の合算。いずれかのソースが失敗した場合は `total=null`、`has_next` フラグで判定する。
**制約**: 複数ソースの横断ページネーションは「各ソースを並列ページング」する設計であり、統合ソートやデデュープは将来課題とする。
