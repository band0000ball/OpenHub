# 外部API連携レジリエンスパターン

## 1. 防御的レスポンスパース

外部APIの `response.json()` 呼び出し前に必ずチェックする：

1. **ステータスコード** — 2xx であることを検証
2. **Content-Type** — `application/json` であることを検証
3. **空ボディ** — レスポンスボディが空でないことを検証

外部APIは仕様通りにレスポンスを返す保証がない。メンテナンス中にHTMLを返す、レート制限時に空レスポンスを返す等のケースが実際に発生する。

> 内部API（自チーム管理）には不要。外部API（管理外）のみ適用。

## 2. 横断検索の耐障害設計

複数データソースを横断検索するループでは、個別ソースのエラーを捕捉し「そのソースをスキップして次へ進む」設計にする。

```
results = []
for source in sources:
    try:
        results += source.search(query)
    except (Timeout, AuthError, UpstreamError):
        log.warning(f"{source.name} skipped")
        continue
return results
```

- 1ソースの障害が全体を止めてはならない
- スキップしたソースはレスポンスに含めてユーザーに通知する
- 全ソースが失敗した場合は明示的にエラーを返す

## 3. 認証状態変化時のキャッシュ無効化

認証状態（APIキー登録・ログイン・ログアウト等）が変化するエンドポイントでは、キャッシュ無効化を必ず実装する。

**問題**: 認証前の空結果がキャッシュに残ると、認証後も古い結果が返される（キャッシュポイズニング）。

**対策**:
- 認証状態をキャッシュキーに含める
- または認証状態変化時にキャッシュを明示的にパージする
- 認証関連エンドポイントのレスポンスは `no-store` にする
