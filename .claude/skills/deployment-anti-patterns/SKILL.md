# デプロイ・ブランチ管理アンチパターン集

## 1. 二重デプロイの順序誤り

フロントエンド（自動デプロイ）とバックエンド（手動デプロイ）が分離している場合に、バックエンドのデプロイを忘れる、またはフロントエンドより後にデプロイする。

**問題**: フロントエンドが新しい API を呼び出すが、バックエンドは旧バージョンのまま → ユーザーに TypeError や 500 エラー。

**デプロイ順序（厳守）**:
1. バックエンドを先にデプロイ（例: `sam deploy`）
2. バックエンドの動作を確認
3. フロントエンドのビルド・デプロイ（例: Amplify 自動ビルド）
4. 統合動作を確認

**予防策**:
- デプロイチェックリストを作成し順序を明文化
- CI/CD パイプラインでバックエンド→フロントエンドの順序を自動化
- PR にフロント/バックどちらの変更を含むかラベルを付ける

## 2. マージ済みブランチへの誤コミット

PR マージ後に同じブランチで作業を継続し、マージ済み PR にコミットが積み上がる。

**対策**: Ship / マージ後は以下を必ず実行：
1. `gh pr view --json state` でブランチの PR 状態を確認
2. マージ済みであれば新しいブランチを切る
3. 新しいブランチで次の作業を開始

**予防策**:
- PR マージ後にローカルブランチを削除する習慣をつける
- CI/CD でマージ済みブランチへの push を警告する仕組みを導入
