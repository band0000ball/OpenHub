# Plan Engineering Review

## 概要

実装計画（Think フェーズの成果物）を実装前にレビューし、テスト計画・完了定義（DoD）・リスクを評価するプロセス。スコア基準で PLAN LOCK（実装着手許可）または PLAN REVISION REQUIRED（再検討要求）を判定する。

## 評価軸

| 軸 | 評価内容 |
|----|---------|
| 設計の妥当性 | アーキテクチャ、技術選定、スコープの適切さ |
| テスト計画 | テスト種類、カバレッジ目標、テストシナリオの網羅性 |
| 実装リスク | 技術的リスク、外部依存、パフォーマンスリスク |
| DoD（完了定義） | 完了条件の明確さ、測定可能性 |

## スコア基準

- **8-10/10**: PLAN LOCK — 実装着手可
- **6-7/10**: PLAN LOCK（条件付き） — HIGH 指摘事項を Build 中に対処する条件で着手可
- **4-5/10**: PLAN REVISION REQUIRED — 指摘事項を修正してから再レビュー
- **1-3/10**: PLAN REJECTED — 根本的な再設計が必要

## 指摘事項の重要度

- **CRITICAL**: 実装前に必ず解決。未解決のまま着手不可
- **HIGH**: Build 中に対処する条件で着手可
- **MEDIUM**: 実装中に改善を推奨
- **LOW**: 次回スプリントで対応可

## プロセス

1. **Think フェーズの成果物を入力とする** — 設計ドキュメント、スコープ定義、技術選定理由
2. **4軸で評価する** — 各軸にスコアを付与
3. **指摘事項を列挙する** — 重要度別に分類
4. **判定を出す** — PLAN LOCK / REVISION REQUIRED / REJECTED
5. **REVISION の場合は修正後に再レビュー**

## UI ありスプリントの場合

- Engineering Review と Design Review を同時に実施する
- Design Review では追加でカラートークン、レスポンシブ、アクセシビリティを評価する

## 適用タイミング

- 全フェーズの Think → Build の間に実施
- 特に新機能追加、アーキテクチャ変更、外部API統合の前は必須
