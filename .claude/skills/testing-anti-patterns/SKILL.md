# テスティングアンチパターン集

## 1. Singleton 追加時のテスト fixture 更新漏れ

新しい状態保持 singleton（キャッシュ、ストア、レジストリ等）を追加した際に、テストの autouse fixture にリセット処理を追加し忘れる。

**症状**:
- テスト単独実行は成功、一括実行で失敗
- テスト実行順序によって結果が変わる

**対策**: singleton を追加したら必ずセットで行う：
1. singleton にリセット関数を実装
2. テストの autouse fixture にリセット呼び出しを追加
3. 全テスト一括実行で順序依存がないことを確認

## 2. E2E テストの静的シェルのみ検証

E2E テストで静的シェル（見出し、ナビゲーション、検索バー等）のみを検証し、実装した UI 要素の実際の動作結果を検証しない。

**問題**: テストは全通過するが、本番ではデータ取得エラーやレンダリング不具合がユーザーに表示される。

**対策**: 動的コンテンツを含むページでは、実装した UI 要素が正しく機能しているかまで検証する：
- 実装した UI 要素の表示内容（リスト項目、テーブル行、カード等が期待通りレンダリングされているか）
- ユーザー操作に対するインタラクション結果（フィルタ適用後の絞り込み、ソート後の並び順等）
- 空状態・エラー状態のフォールバック UI

**チェックリスト**:
- [ ] データ一覧: 少なくとも1件の実データ項目が表示されているか
- [ ] 詳細ページ: 実装したフィールド（タイトル、説明、メタデータ等）が表示されているか
- [ ] 検索・フィルタ: 操作結果が UI に反映されているか
- [ ] フォーム: 送信後の状態遷移が正しいか
