# DocDD リバースシンク

## 概要

Ship（PR マージ）後に、ドキュメント（CHANGELOG・README・DESIGN.md）を実装の実態に合わせて同期するプロセス。DocDD（Documentation-Driven Development）の逆方向フロー。

## 目的

- 実装とドキュメントの乖離を防ぐ
- DESIGN.md に記載のない実装を検出し、設計ドキュメントを最新化する
- CHANGELOG を常に正確に保つ

## 手順

### 1. 差分の確認

Ship した PR の変更内容を確認する：
- `git diff [base]...HEAD` で全変更を把握
- 新規追加された機能、変更された振る舞い、削除された機能を列挙

### 2. CHANGELOG の更新

- Added / Changed / Fixed / Removed の適切なセクションにエントリを追加
- 日付とフェーズ番号を明記

### 3. README の更新

- 新機能の説明を追加
- 環境変数や設定値の変更を反映
- セットアップ手順の変更を反映

### 4. DESIGN.md の乖離チェック

- 実装内容が DESIGN.md に記載されているか確認
- 記載のない実装を検出した場合：
  - 意図的な追加 → DESIGN.md に追記
  - 設計からの逸脱 → チームで議論

### 5. 乖離レポート

乖離が見つかった場合、以下の形式で記録：
- 乖離箇所（ファイル・セクション）
- 実装の実態
- ドキュメントの記載内容
- 対応（追記 / 修正 / 議論が必要）

## 適用タイミング

- 毎回の Ship / PR マージ後
- 特に機能追加・API 変更・設定変更を含む PR の後は必須
