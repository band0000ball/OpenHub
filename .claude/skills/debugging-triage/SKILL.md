# 障害トリアージ・調査パターン集

## 1. 認証障害の連鎖影響 — 認証優先トリアージ

認証基盤の障害は依存する全機能を連鎖的に壊す。

**連鎖パターン**:
1. 認証が壊れる
2. API 呼び出しが 401/403 で失敗
3. データ保存が 502 で失敗
4. データ取得が空結果を返す
5. UI がエラー表示になる

個別のエラーとして見ると原因がバラバラに見えるが、実際には単一の認証障害が根因。

**トリアージ手順**:
1. 複数エラーが同時発生 → まず認証フローを確認
2. 認証が壊れていれば、他のエラーは無視して認証を先に修正
3. 認証修正後に残るエラーのみを個別対応

**適用場面**:
- 本番で複数の異なるエラーが同時に報告された場合
- デプロイ後に複数の機能が同時に壊れた場合
- 認証ミドルウェアや認証設定を変更した後

## 2. 調査ファーストアプローチ

原因が不明な問題は「修正」ではなく「調査」を先行する。

**修正先行のリスク**:
- 問題を正しく特定できず的外れな修正をする
- 実際には問題がない場合に不要な修正でコードを複雑化する
- 修正の副作用で新たな問題を引き起こす

**進め方**:
1. **調査スコープを定義** — 何を確認すれば問題の有無が判断できるか
2. **観測データを収集** — 実 API レスポンス、ログ、再現手順
3. **判定** — 問題あり（修正へ） / 問題なし（調査レポートで完了） / 判断不可（追加調査）
4. **結果を記録** — 問題なしの場合も記録する（同じ調査の繰り返しを防ぐ）

修正不可の場合は調査レポートで完了とする。「調査したが問題なし」も立派な成果。
