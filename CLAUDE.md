# CLAUDE.md

## アイデンティティと原則

このプロジェクトは **NFD（Nurture-First Development）** に基づいて運用します。
エージェントは対話を通じて育成され、知識は結晶化セッションで構造化されます。
用途（開発・データ分析・リサーチ・資料作成）を問わず **gstack スプリントで進めます**。
**複雑化しない**：1フェーズが複雑になる場合はタスクを分割して再スタートします。

## ワークスペース別の動作ルール

### Nurturing ワークスペース（日常業務）

gstack スプリントに Reflexion ループを組み込んで進めます。

| フェーズ | 役割 | 共通フォーカス | 開発 | 分析・リサーチ・資料 |
|---------|------|--------------|------|-------------------|
| Think   | Actor | 問いの再定義・目的の明確化 | デザインドック | 課題定義・アウトライン |
| Plan    | Actor | 構造・手法・設計の確定 | 仕様書・ドメイン設計 | 調査設計・構成設計 |
| Build   | Actor | 成果物の生成 | 実装コード | 分析・執筆・情報収集 |
| Review  | Evaluator | 品質・整合性の確認 | コード↔仕様の整合 | 論理・手法・正確性 |
| Test    | Evaluator | 検証 | テストコード実行 | データ検証・通読確認 |
| Ship    | Actor | 納品・提出 | PR 自動作成 | レポート・資料の提出 |
| Reflect | Reflector | 振り返り・ドキュメント同期 | `/document-release` | プロセス改善メモ |

**DocDD（ドキュメント駆動開発）の適用範囲：**

| 用途 | 適用レベル | 原則 |
|------|----------|------|
| 開発 | 完全適用 | ドキュメント → コード生成。コードの直接修正は原則禁止 |
| 資料作成 | 任意適用 | アウトライン → コンテンツ生成 |
| データ分析・リサーチ | 適用しない | gstack スプリントのみ |

- **Reflector ステップ**：Review・Test・Reflect フェーズ終了時に必ずログを記録する
- テストコードの作成は必須（開発タスクのみ）。テスト全通過後、自動で PR を作成する
- スキル改善要望は `update/[YYYYMMDD-HHmmss].md` に記録する

### Surgical ワークスペース（週1回・結晶化専用）

Reflexion のマクロ Reflector として機能します。

- 実装・コード変更は禁止
- ログ分析 → パターン抽出 → 脱文脈化 → 人間バリデーション → Skills 更新のみ実施する
- 成果物：Skills・エラーパターンライブラリ・意思決定ツリー・用語グロサリー・アンチパターン集

## ログフォーマット（必須）

ファイル：`memory/[YYYYMMDD-HHmmss].log`

```
[タグ] タイトル

## 提案内容

## 提案理由

## 判断理由

## 詳細
```

タグ一覧（4結晶化セッションごとに見直す）：

| タグ | 用途                 | 主な発生フェーズ |
|------|--------------------|----------------|
| `[DECISION]` | 意思決定               | Think / Plan |
| `[INSIGHT]` | 方針変換・新仕様の発見        | Review / Reflect |
| `[ERROR]` | 認識違い・仕様の曖昧さ・エラー・障害 | Review / Test |

## 成果物ディレクトリ

```
memory/                        # 経験層ログ
update/                        # スキル改善要望
patterns/

  glossary/                    # 用語グロサリー
  error/

    anti-pattern/              # アンチパターン集
```

## テストフレームワーク（開発タスク）

| 言語 | フレームワーク | テストファイル命名規則 |
|------|--------------|-------------------|
| Python | pytest | `test_*.py` または `*_test.py` |
| C# | xUnit.net | `*Tests.cs` |
| JS/TS | Vitest（Vite系）/ Jest（その他） | `*.test.ts` / `*.spec.ts` |

- テストは実装ファイルと同ディレクトリ、または `tests/` サブディレクトリに配置する
- PR 作成前に全テストが自動実行され、失敗した場合は PR 作成がブロックされる

## フェーズ × スキル早見表

| フェーズ | 呼ぶスキル | 条件 |
|---------|----------|------|
| Think   | `/office-hours` | 常時。スコープ確認→`/plan-ceo-review`も |
| Plan    | `/plan-eng-review` | 常時。UI有り→`/plan-design-review` `/design-consultation`も |
| Build   | `/investigate` | バグ発生時。本番→`/careful`。範囲制限→`/freeze` |
| Review  | `/review` | 常時。UI有り→`/design-review`も |
| Test    | `/qa` | Web有り。報告のみ→`/qa-only`。失敗時→`/investigate` |
| Ship    | `/ship` | 常時。本番デプロイ→`/careful`も |
| Reflect | `/document-release` | 常時。週次→`/retro`も |

## 禁止事項

- テストなしの実装完了宣言（開発タスク）
- ログなしの意思決定・提案
- Surgical ワークスペースでの実装作業
- 開発タスクでのコードの直接修正（ドキュメント経由を優先する）
- 1フェーズに複数の責務を詰め込む（複雑化したらタスク分割）
